# DB Connect 安全说明

## 凭据与敏感数据存储

- **连接配置加密**：所有已保存的连接配置会序列化为 JSON 后，使用 AES-256-GCM 进行加密，密钥存储在操作系统提供的钥匙串中（通过 `keyring` 库访问）。若配置了数据库 SSL/TLS（CA 路径、PKCS#12 路径等），应用仅保存这些**文件路径**字符串；证书与私钥文件本身仍由用户保管在磁盘上，请妥善设置文件权限。
- **密钥管理**：应用不会在磁盘上直接保存加密密钥，仅通过系统安全存储读取；密钥生成后长期复用。
- **文件位置**：加密后的连接配置存储在应用数据目录下的 `connections.json` 文件中，表列设置存储在 `table-column-settings.json` 中，SSH known_hosts 信息存储在用户主目录下的 `~/.db-connect/ssh_known_hosts.json` 文件中。
- **权限控制**：在 Unix 平台上，上述敏感目录会被设置为 `0o700`、敏感文件为 `0o600`，确保只有当前用户可读写（Windows 平台沿用默认 ACL 模型）。

## SQL 执行模型

- **表数据 CRUD**：通过参数化查询和安全转义（`esc_id` / `esc_str`），后端不会将用户提供的数据直接拼接进 SQL 字符串中，以降低 SQL 注入风险。
- **WHERE 子句**：前端的 Where 构建器生成筛选条件，后端额外使用 `validate_where_clause` 进行校验，拒绝包含分号、SQL 注释以及 `UNION`、`INTO OUTFILE`、`LOAD_FILE`、`EXECUTE`、`LOAD DATA` 等高危关键字的子句，进一步防止注入尝试。
- **SQL 编辑器（execute_sql）**：
  - 作为桌面 SQL 客户端特性，SQL 编辑器会按用户输入执行完整 SQL 语句，包括 `SELECT`、DML、DDL 等，适用于可信本地开发与运维场景。
  - 为防止异常或恶意的超长输入导致资源耗尽，单条 SQL 文本在后端具有长度上限（约 1MB），超出将被拒绝执行。
  - `SELECT`、`SHOW`、`EXPLAIN` 等返回结果集的语句，单次返回行数上限为 **100,000**（与 CSV 导出上限一致），超出将报错并提示使用 `LIMIT`；可降低客户端内存与 IPC 压力。
  - 可选 **只读连接**：启用后禁止除查询类语句（MySQL 额外允许 `USE` 切库）以外的 SQL 编辑器写语句，并阻止表结构编辑、数据写入、索引/触发器/外键/例程变更、SQL 文件导入等写类后端命令。
  - 建议仅在**可信环境**下使用本应用，不要将其前端或后端暴露为远程多租户服务。

## 权限与隔离

- **应用模型**：本应用为本地桌面客户端，不内置多用户登录或权限隔离；能够登录当前操作系统帐户的用户，即默认拥有应用内所有已保存连接的访问能力。
- **Tauri 权限**：前端仅通过 Tauri IPC 调用受限的后端命令；默认 capability 不允许任意文件系统或 shell 访问，仅开放必要的核心与剪贴板写入权限。

## 依赖安全

- 建议在本地或 CI 中定期执行以下命令，检查第三方依赖安全性：
  - Rust 依赖审计：`npm run audit:rust`（依赖 `cargo-audit` 已安装）
  - 前端依赖审计：`npm run audit:npm`
- 如发现高危漏洞，应优先升级依赖或参考公告采取缓解措施。

## 使用建议

- 仅在受信任的本机环境中使用 DB Connect，不要直接对外暴露其 IPC 或打包后的应用。
- 不要在共享账号或未加保护的系统帐户下保存敏感生产连接配置。
- 对于生产数据库，建议配合数据库本身的权限管理（最小权限原则），为 GUI 客户端连接分配专门的账号。

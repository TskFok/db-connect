# 数据库结构同步进度与 NULL 默认值展示设计

## 背景与目标

当前数据库结构同步在一次 `execute_database_sync` 调用内依次完成执行前结构校验、DDL 执行和执行后重新对比。前端只能获得 `executing` 布尔状态，因此耗时较长时只显示按钮加载动画，用户无法判断当前阶段或已完成数量。

数据库对比结果的默认值展示还存在一个独立问题：统一快照用 `null` 表示数据库元数据返回的 NULL 默认值，前端格式化函数把 `null` 转为空字符串，最终显示为“默认值=”。本次将其明确显示为“默认值=NULL”。

本次目标：

- 为完整同步执行流程提供实时、可访问的进度反馈。
- DDL 执行阶段显示真实的已完成语句数和总语句数。
- 不改变现有无状态两阶段计划、指纹校验、删除保护和首个失败即停止语义。
- 修复源端与目标端字段默认值为 `null` 时的空白展示。

## 范围

### 包含

- 执行前结构校验阶段的不确定进度提示。
- DDL 执行阶段按 SQL 语句上报确定进度。
- 执行后重新对比阶段的不确定进度提示。
- 同步预览弹窗中的进度条、阶段文本和语句计数。
- 通过计划指纹过滤进度事件，避免不同同步请求之间串扰。
- 监听失败时退化为现有加载状态，不影响同步命令执行。
- 数据库对比字段默认值 `null` 显示为 `NULL`。

### 不包含

- 不支持取消、暂停、恢复同步。
- 不引入后端有状态任务、轮询接口或持久化进度。
- 不估算单条 DDL 的数据库内部执行百分比。
- 不改变 `null` 在快照模型中的含义，也不尝试区分数据库元数据无法区分的“无默认约束”和“DEFAULT NULL”。
- 不改变同步 SQL、操作排序或失败结果契约。

## 方案选择

采用后端实时事件方案。

未采用的方案：

- 前端模拟百分比无法反映真实执行位置，长时间 DDL 会给出误导反馈。
- 后端任务轮询需要任务注册、生命周期和清理机制，复杂度超过本次需求。

## 进度契约

新增前后端一致的同步进度负载：

```text
DatabaseSyncProgress
  plan_fingerprint: string
  phase: validating | executing | refreshing
  current: number
  total: number
```

事件名固定为 `database-sync-progress`。

- `validating`：执行命令开始、重新建立临时连接并生成当前计划时发送；`current = 0`、`total = 0`。
- `executing`：指纹及执行保护校验通过后发送初始事件，`current = 0`、`total` 为计划内全部 SQL 语句数；每条 SQL 成功后发送一次，`current` 增加一。
- `refreshing`：全部 DDL 成功、开始重新读取两端结构时发送；沿用已完成语句数与总语句数。

执行失败时不伪造完成事件。结构化执行结果到达后，前端切换为现有成功、部分成功或失败结果视图。

事件只包含计划指纹、阶段和计数，不包含 SQL、连接配置或凭据。事件发送失败不导致同步失败。

## 后端数据流

`execute_database_sync` 接收现有 `AppHandle`，在进入执行生命周期前发送 `validating`。计划重新生成并通过指纹、阻塞项和删除保护校验后，统计 `preview.operations[*].sql` 的总数并发送 `executing 0/total`。

现有逐语句执行器增加一个进度回调。每条 SQL 成功后记录 `DatabaseSyncStatementSuccess`，随后调用回调上报当前计数；失败时继续沿用首个失败停止和部分结果构造逻辑。

全部 SQL 成功后，在重新读取源端、目标端结构之前发送 `refreshing`。重新对比完成或清理异常仍使用现有执行结果表达。

纯执行器测试通过注入回调验证计数，不依赖 Tauri 运行时。命令层负责把回调转换为 `AppHandle.emit`。

## 前端交互

`DatabaseCompareModal` 在调用 `executeDatabaseSync` 前注册 `database-sync-progress` 监听，并重置旧进度。收到事件后，仅当 `plan_fingerprint` 等于当前确认计划时更新状态。

执行结束、报错、预览清理或组件卸载时移除监听并清空进度，防止过期事件更新新计划。若监听注册失败，命令照常执行，弹窗保留现有按钮加载与“正在执行”状态。

`DatabaseSyncPreviewModal` 在执行中展示 Ant Design `Progress`：

- 校验阶段：不确定进度，文案“正在校验源端与目标端结构”。
- 执行阶段：确定进度，按 `current / total` 计算百分比，文案“正在执行 DDL，已完成 X / Y 条语句”。
- 刷新阶段：显示不确定进度，文案“DDL 已执行完成，正在刷新结构对比”，避免把 DDL 的 100% 误解为整个流程已经结束。
- 尚未收到事件或监听不可用：不确定进度，文案沿用“正在执行数据库结构同步”。

进度区域使用 `role=status` 与 `aria-live=polite`，现有执行锁、禁止关闭和禁止重复提交行为保持不变。

## NULL 默认值展示

在 `formatSnapshotValue` 中对 `default_value` 字段单独处理：值为 `null` 时返回字面量 `NULL`。其他布尔、数字和字符串字段继续沿用现有格式化逻辑。

因此单侧字段完整属性会从：

```text
默认值=
```

变为：

```text
默认值=NULL
```

默认值是非空表达式时保持原样。

## 错误与并发处理

- 监听注册失败不阻断命令，不向用户制造同步失败。
- 全局 Tauri 事件通过计划指纹过滤；旧计划事件不能更新当前进度。
- 指纹校验失败、阻塞项或删除保护失败时，命令返回现有错误，前端移除监听并关闭执行状态。
- DDL 失败时最后一个进度值只代表已成功语句数，执行结果负责展示失败位置。
- 事件发送不参与数据库操作结果，发送失败被忽略。

## 测试策略

### 前端

- `databaseCompare` 纯函数测试：`default_value: null` 输出 `默认值=NULL`；非空默认值保持不变。
- 同步进度纯函数测试：确定进度百分比边界和阶段文案。
- `DatabaseSyncPreviewModal` 测试：校验、执行、刷新和无事件退化状态。
- `DatabaseCompareModal` 测试：执行前注册监听、按计划指纹过滤、完成与异常后清理监听。

### 后端

- 逐语句执行器测试：初始总数正确，每条成功语句按顺序上报。
- 首条失败与部分失败测试：只上报实际成功数量，不继续上报未执行语句。
- 命令源码或边界测试：`database-sync-progress` 事件已接入校验、执行和刷新阶段。

## 验收标准

- 点击确认执行后立即看到校验阶段进度提示。
- 进入 DDL 阶段后显示真实的 `X / Y` 语句进度，成功执行一条更新一次。
- DDL 全部完成后显示正在刷新对比，直至结果返回。
- 同步成功、部分失败、首条失败和结构漂移的既有语义不变。
- 目标端默认值为 `null` 时显示“默认值=NULL”，不再显示“默认值=”。
- 不在循环遍历中新增 SQL 查询。
